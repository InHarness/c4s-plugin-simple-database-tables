# Brief: 0.1.1 → 0.1.2

## What changed

This release tightens one behavioral rule for the frontend chip/embed renderers: the
hand-written `renderChip` / `renderCard` / `renderRow` markup must now style itself using the
host's token-bridge CSS variables instead of emitting bare, unstyled elements.

Previously the spec only said these renderers were "pure-presentational with the host-resolver"
without pinning down how they should look. The gap meant a renderer could emit raw `<div>`/`<span>`
markup with no styling, which renders **"naked"** inside the host shell — an embedded element
(e.g. a `database-table` single-element embed) would show no background, borders, or text framing.

### The new rule

`renderCard` and `renderRow` (and `renderChip`) MUST style themselves with the host **`--c-*` CSS
variables** exposed by the token bridge — the same `c4s-paper-terra` design-system variables already
consumed by the plugin's `frontend.routes` list route. Concretely, use:

- `var(--c-bg)`   — background
- `var(--c-fg)`   — foreground / text
- `var(--c-muted)` — secondary / muted text
- `var(--c-border)` — borders / dividers

These must be referenced via the element's `style` or `className`. Do **not** hard-code colors —
using the tokens is what keeps the host's theme modes (light/dark) working. A renderer that emits
bare elements with no `--c-*` reference is now considered non-conformant.

This affects only the per-element chip/card/row markup. Note that `renderRow` still renders a single
row only — there is no list-header/container slot in `FrontendModule`; the list header
(`EntityListHeader`) remains the responsibility of the plugin's own `frontend.routes` list route, not
`renderRow`. That boundary is unchanged.

### New acceptance criterion

A new active requirement-style AC formalizes the rule and is verified against the two UI views it
governs — `database-table-detail-panel` and `database-table-list` (tag: `frontend`):

> renderCard and renderRow style themselves with the host token-bridge `--c-*` CSS vars (e.g.
> `var(--c-bg)`, `var(--c-fg)`, `var(--c-muted)`, `var(--c-border)`) rather than emitting unstyled
> bare markup, so single-element embeds are not rendered naked in the host.

## For implementers

Target the chip/embed renderers in the frontend module (the `renderChip` / `renderCard` / `renderRow`
functions described in the "L8 — Chips" section of `frontend.md`):

1. In each of `renderCard`, `renderRow`, and `renderChip`, apply inline `style` (or a `className`
   that resolves to these) using the host token variables. For example, a card container should set
   `background: var(--c-bg)`, `color: var(--c-fg)`, and `border: 1px solid var(--c-border)`, and use
   `var(--c-muted)` for secondary text.
2. Reuse the exact same `--c-*` token set already consumed in the `frontend.routes` list route so the
   embeds match the rest of the host theme.
3. Remove any hard-coded color literals from these renderers; rely on the tokens so light/dark theme
   modes keep working.
4. Verify against the `database-table-list` and `database-table-detail-panel` views: embedded
   single-element renders should show proper background/border/text framing rather than naked markup.
