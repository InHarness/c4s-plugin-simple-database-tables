# Brief: v0.1.0 → 0.1.1

This release tightens the **frontend spec** of the `simple-database-tables` plugin in two
related ways: (1) it makes every plugin data fetch **project-scoped** and (2) it rebuilds the
list view on top of the **Host UI Kit** (`EntityListRow`, `TagFilterBar`, `EntityListHeader`).
Three new acceptance criteria pin these down as observable contracts. The backend, manifest,
serialization, and the `database-table` entity itself are unchanged.

## 1. Project-scoped data fetches are now mandatory (the headline change)

Previously the spec said only that data resolution is host-injected via the `useGetBySlug`
resolver (`use__EntityName__BySlug`). It did not say who builds the URL. Now the rule is explicit:

- **The plugin composes the URL, the host invokes the resolver.** The host does **not** prefix
  the plugin's data fetches.
- Every plugin data fetch MUST build a URL of the form
  `/api/projects/<id>/database-tables/...`, where `<id> = window.__C4S_PROJECT__.id`.
- A bare `/api/...` request (without the `/api/projects/<id>` prefix) resolves to **404** — a
  loud failure, never a silent degradation.

Key distinctions the spec calls out:

- **Routing ≠ fetch.** The host auto-applies a basepath **only** to client navigation (the route
  tree mounted through `host.mountFrontend`). Data fetching is a separate path and the plugin
  must add the project prefix itself.
- **No host helper yet.** There is no host-provided `projectId` / `apiFetch` accessor. The plugin
  reads the id directly with the fallback:
  ```js
  const projectId = window.__C4S_PROJECT__?.id ?? 'default';
  ```
  If such a helper later ships, it would land on the versioned `hostApiVersion` surface.
- **Timing.** `window.__C4S_PROJECT__` is injected before React mounts, so the id is available
  the moment a chip, card, or detail panel renders.
- The rule holds regardless of the resolver's name — `useGetBySlug` is the M13 data-resolver
  concept, not an M33 contract, but any hook that fetches plugin data must apply the prefix.

Both UI-view specs now carry this constraint:

- **Database Table Detail Panel** — its self-fetch (via the registered `useGetBySlug`) must now
  compose `/api/projects/<id>/database-tables/...` from `window.__C4S_PROJECT__.id`; the host does
  not prefix it.
- **Database Table List** — same project-scoped fetch requirement added.

## 2. List view rebuilt on the Host UI Kit

The list view changes from hand-written row markup to the Host UI Kit (L12 / M34) components,
imported from `@c4s/plugin-runtime/ui` (re-exported through `src/host.ts`).

**Row rendering.** `renderRow` is now rendered through **`EntityListRow`** instead of bespoke
markup. The previous shape was "name + a column-count badge + description". The new mapping onto
`EntityListRow` props:

- `title` (required) = entity name
- `meta` = column count
- `subtitle` = description
- remaining `ReactNode` slots: `icon?`, `actions?`, `children?`

**List route composition.** `FrontendModule` still has **no** list-header/container slot, so
`EntityListHeader` is never inside `renderRow`. Instead, the plugin's own list route
(`frontend.routes`) composes the full native sequence:

```
EntityListHeader → search → TagFilterBar → EntityListRow
```

- `TagFilterBar` props: `tags`, `selected`, `onToggle`, `mode`, `onModeChange?`. It renders each
  tag chip via the existing `Badge` component.
- The "N results" counter is the **`count`** prop on `EntityListHeader` — there is **no** separate
  `CountBadge`.

The slash command `/database-table` (slug-aligned, not `/dbtable`) is unchanged.

**Detail panel** is otherwise unchanged from v0.1.0 — `DetailPanelShell` (frame + breadcrumb, no
`title` prop), `FieldGrid` / `FieldRow` (value as `children`, not a `value` prop), token bridge
exposing `c4s-paper-terra` tokens as `--c-*` CSS vars — except for the added project-scoped fetch
requirement above.

## 3. Component stability tiers (new)

The Host UI Kit is now explicitly split into two tiers, which gates how you may adopt each
component:

- **`stable`** — frozen, part of `hostApiVersion`; a breaking change requires a major bump plus
  `migrations[]`. Components: `EntityListHeader`, `DetailPanelShell`, `FieldGrid`, `FieldRow`.
- **`experimental`** — props may change without a major bump, ungated, opt-in. Components:
  `TagFilterBar`, `EntityListRow`. Adopting them is a deliberate decision that accepts an
  unstable contract.

Note that the list view's new dependencies (`EntityListRow`, `TagFilterBar`) are both
**experimental-tier** — adopting them accepts an unstable contract.

**Build/dependency requirements are unchanged:** `@c4s/plugin-runtime/ui` must remain in Vite's
`rollupOptions.external`; no new `peerDependencies` key (the subpath ships with the existing
`@c4s/plugin-runtime` peer); manifest / `contributes` / `capabilities` unchanged.

## 4. New acceptance criteria

Three AC were added to lock the above as the observable contract:

- **(requirement)** "The frontend database-tables module reads the project id from
  `window.__C4S_PROJECT__.id` and prefixes every project-scoped fetch as
  `/api/projects/<id>/database-tables/...`" — verifies both the list view and the detail panel.
- **(edge-case)** "A plugin backend call made without the `/api/projects/<id>` prefix returns
  404 — a loud failure, never a silent degradation."
- **(requirement)** "The list view renders rows via `EntityListRow`, offers a tag filter via
  `TagFilterBar`, and renders the result count through the `count` prop on `EntityListHeader`." —
  verifies the list view.

## For implementers

Concrete edit targets in the implementing plugin repo (bootstrapped from
`InHarness/c4s-plugin-scaffold`):

1. **Project-scoped fetch helper.** Wherever the frontend builds data-fetch URLs (the
   `useGetBySlug` / `use__EntityName__BySlug` resolver implementation for `database-table`),
   prepend `/api/projects/${window.__C4S_PROJECT__?.id ?? 'default'}/database-tables/...`. Do NOT
   rely on the host basepath — that applies to navigation only. Both the list view and the detail
   panel fetches must be updated.
2. **List row component.** Replace the hand-written `renderRow` markup with the kit's
   `EntityListRow`, imported from `@c4s/plugin-runtime/ui` (re-exported via `src/host.ts`). Map:
   `title={name}`, `meta={columnCount}`, `subtitle={description}`.
3. **List route.** In `frontend.routes`, build the sequence `EntityListHeader` (pass the result
   count via its `count` prop) → search → `TagFilterBar` (`tags`, `selected`, `onToggle`, `mode`,
   `onModeChange?`; chips via `Badge`) → `EntityListRow`. Keep `EntityListHeader` out of
   `renderRow` (no list-header slot on `FrontendModule`).
4. **Build config.** Ensure `@c4s/plugin-runtime/ui` stays in Vite `rollupOptions.external`. No
   change to `peerDependencies`, manifest, `contributes`, or `capabilities`.
5. **Tier awareness.** `TagFilterBar` and `EntityListRow` are experimental-tier — expect their
   props to change without a major host bump.
